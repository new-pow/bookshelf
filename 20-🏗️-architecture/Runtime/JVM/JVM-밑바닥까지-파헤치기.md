## 2.2 런타임 데이터 영역
- 자바 프로그램을 실행하는 동안 필요한 메모리를 몇 개의 데이터 영역으로 나눠 관리
- 각 영역들은 목적/생성시점/삭제시점이 있다.

| ----------- | ------------------------------------------------------------ | --------------------- | --------------------- | ---------------- |
| 프로그램 카운터    | 현재 실행 중인 JVM 바이트코드의 **명령어 주소 저장**. 스레드가 어디까지 실행했는지 추적        | 스레드 생성 시              | 스레드 종료 시              | **스레드 프라이빗 메모리** |

### 프로그램 카운터
- 현재 실행중인 스레드의 바이트코드 줄 번호 표시기
- 프로그램 제어 흐름, 분기, 순환, 점프 등을표현한다.
- JVM에서 멀티스레딩은 CPU 코어를 여러 스레드가 교대로 사용하는 방식으로 구현되어있다.
	- 특정 시각에 각 코어는 한 스레드의 명령어만 실행하게 된다.
	- 즉, 스레드 전환 후 이전에 실행하다 멈춘 지점을 정확하게 복원하려면 각 스레드는 고유한 프로그램 카운터가 필요하다.
- 스레드 프라이빗메모리
	- 스레드카운터는 서로 영향을 주지 않는 독립된 영역에 저장된다.
- 자바 메서드를 실행중일때는, 실행 중인 바이트코드 명령어의 주소가 프로그램 카운터에 기록됨.
- 네이티브메서드를 실행중일때는 프로그램카운터값은 `Undefined` 입니다.

### 자바 가상 머신 스택
- 스레드 프라이빗하다
- 연결된 스레드와 수명주기를 같이한다.
- 각 메서드가 호출될때마다 JVM은 스택 프레임을 만들어, 정보를 저장하고 스택 프레임을 가상머신 스택에 push 하고 메서드가 끝나면 pop 한다.
	- 지역변수테이블
	- 피연산자 스택
	- 동적 링크
	- 메서드 반환값 등의 정보
- C, C++ 프로그램의 메모리 구조보다 복잡하게 되어있다.
- 지역변수테이블
	- 기본 데이터 타입, 객체 참조, 반환 주소 타입을 저장한다
	- 이 데이터 타입들을 저장하는 공간을 지역변수슬롯 이라고 한다.
		- 이 슬롯은 기본 32bit. double 타입과같이 길이가 64bits 라면 슬롯을 2개 할당한다.
	- 컴파일 타임에 지역변수 테이블을 구성하는데 필요한 데이터 공간(즉, 슬롯 갯수)을  모두 할당하여 결정한다.
- 스택 메모리 영역에서는 두 가지 오류가 발생할 수 있다.
	- StackOverflowError: 스택 깊이가 가상 머신이 허용하는 깊이보다 클 경우
	- OutOfMemoryError: 스택 용량을 동적으로 확장할 수 있는 경우에, 스택을 확장하려는 시점에 여유 메모리가 충분하지 않을 경우.

### 메서드 영역
- 모든 스레드가 공유
- JVM이 읽어들인 타입 정보, 상수, 정적 변수, JIT 컴파일러가 컴파일한 코드 캐시 등을 저장하는 데 이용
- 이전에는 영구세대 영역에 메서드 영역을 저장했지만, 최근들어 점진적으로 네이티브 메모리에 구현
	- 메서드 영역을 영구 세대에 구현하면 자바 애플리케이션들이 메모리 오버플로를 겪을 가능성이 커졌다.
- JVM spec 에서는 메서드 영역에 제약을 거의 두지 않았다.
	- 연속될 필요가 없고, 크기를 고정할 수 있고-확장할 수도 있음.
	- 가비지 컬렉션을 할필요가 별로 없음 (회수 효과가 매우 작기 때문)
- 메모리를 더이상 할당할 수 없다면 OOM

### 런타임 상수풀
- 동적이다.
- 메서드 영역의 일부
- 클래스 버전, 필드, 메서드, 인터페이스 등 클래스 파일에 포함된 설명 정보에 더해 컴파일타임에 생성된 다양한 리터럴과 심벌 참조가 저장된다.
	- 클래스를 로드할 때 생성된 다양한 리터럴과 심벌 참조가 저장된다.

### 다이렉트 메모리
- 가상 머신 런타임에 속하지 않는다.
- 하지만 JDK 1.4에서 NIO가 조입되면서 채널과 버퍼기반 I/O 메서드가 소개됨
	- 힙이 아닌 메모리를 직접 할당할 수 있는 '네이티브 함수 라이브러리'를 이용하며 이 메모리에 저장되어있는 DirectByteBuffer 객체를 통해 작업을 수행
		- 자바힙과 네이티브 힙 사이에 데이터를 복사해 주고받지 않아도 되어서 일부 시나리오에서 성능을 크게 개선함.
- 물리 메모리를 직접 할당하기 때문에 자바 힙 크기의 제약과 무관하지만, 하부 기기의 총 메모리 용량과 프로세서가 다룰 수 있는 주소 공간을 넘어설 수 없으므로 주의해야한다.
- 사용되는 모든 메모리영역의 합이 물리메로리 한계를 넘어서면 동적 확장을 시도할때 OOM이 발생한다.

## 핫스팟 가상머신에서의 객체 들여다보기
### 객체 생성
1. JVM이 new 명령에 해당하는 바이트 코드 만남
2. 이 명령의 매개변슈가 상수 풀안의 클래스를 가리키는 심벌참조인지 확인
3. 클래스를 가리키고 있다면, 클래스가 로딩-resolve-초기화 되었는지 확인
	1. 만약 안되어있다면 로딩부터 한다
4. 새 객체를 담을 메모리를 자바 힙에 할당
	1. 클래스를 로딩하고나면 어떤 크기가 필요한지 완벽하게 알 수 있다
	2. 사용중인 메모리는 한쪽에, 여유 메모리는 반대편에. 경계에는 포인터가 있다.
	3. 할당하면, 포인터를 객체 크기만큼 이동시킨다 = 포인터 밀치기
	4. 자바 힙이 규칙적이지 않으므로, JVM은 가용 메모리 블록들을 목록으로 관리하여, 객체를 저장할 수 있는 충분한 공간을 찾아 할당 후, 목록 갱신 (이 목록을 free list 라고함.)
	5. 자바 힙이 규칙적인지의 여뷰, GC가 컴팩트를 할 수 있느냐에 따라 다른 방식을 취하기도 함.
5. 할당이 종료되면, JVM은 할당 받은 공간을 데이터 0으로 초기화.
6. JVM은 각 객체에 필요한 설정을 각 객체의 객체 헤더에 저장
	1. 어느 클래스의 인스턴스인지?
	2. 메타 정보는 어떻게 찾는지
	3. 해시코드
	4. GC age
7. 이제 자바 프로그램 관점에서, 생성자 `<init>()` 를 실행하여 개발자 의도대로 초기화

```text
<자바코드>     <바이트코드>
new ---------> new // JVM 관점에서 객체셍성
    ---------> invokespecial // 자바 프로그램 관점에서 객체 생성
```

- 멀티스레딩 환경에서 동시성 문제 해결방식
	- 방법1. 메모리 할당을 동기화하거나
	- 방법2. 스레드마다 다른 메모리 공간을 할당함. 이를 스레드로컬할당버퍼(TLAB) 라고한다.

### 객체에 접근하기
- 자바 프로그램은 스택에 있는 참조 데이터를 통해 힙에 들어 있는 객체들에 접근해 이를 조작
- JVM Spec 은 참조 타입을 단지 '객체를 가리키는 참조' 라고만 정했을 뿐, 구체적인 방식 규정은 없다.
- 주로 핸들 방식, 다이렉트 포인터를 사용해 구현
- 핸들
	- 참조에 '안정적인' 핸들의 주소가 저장됨.
	- 객체의 위치가 바뀌는 상황에서도 참조 자체는 손댈 필요가 없음.
	- 핸들 내의 인스턴트 데이터 포인터만 변경
- 다이렉트 포인터
	- 자바 힙에 위치한 객체에서 인스턴스 데이터 뿐 아니라 타입 데이터에 접근하는 길도 제공
	- 스택에 참조에는 객체 주소가 그대로 저장됨
	- 속도가 빠름. -> 실행 시간에 유리

## 실전: OutOfMemoryError 예외
### 자바 힙 오버플로
- 원인: 객체를 계속 생성하고 그 객체들에 접근할 경로가 살아있는 경우
- ==!: p.78 힙덤프 사례를 볼 수 있음.==
- 대응 방법
	- 메모리 누수
		- 누수된 객체로부터 GC 루트까지의 참조 사슬을 살펴본다.
		- 왜 가비지 컬렉터가 회수 못했는지 확인
	- 만약 메모리 누수가 아닐 경우
		- 더 많은 메모리를 할당할 수 있는지 확인
		- 햅 매개변수설정 `-Xmx`, `-Xms`
		- 코드에서 수명 주기가 너무 길거나 상태를 너무 오래 유지하는 객체 확인
		- 공간 낭비가 심한 데이터 구조를 쓰고 있는지 확인

### 가상머신 스택과 네이티브 메서드 스택 오버플로
- 예외 발생 원인
	- 스레드가 요구하는 스택 깊이가 가상 머신이 허용하는 최대 깊이보다 크면 stackoverflowError
	- 가상 머신이 스택 메모리를 동적으로 확장하는 기능을지원하나, 가용 메모리가 부족해 스택확장이 불가능할경우 OOM
	- 핫스팟에서도 스레드를 계속 만들면 메모리 오버플로를 일으킬 수 있다.
- 보통 운영체제가 각 프로세스에 할당하는 메모리크기가 제한적임.

	- 언제 회수해야 하나?

- 자동화된 GC를 이해해야하는 이유
	- 다양한 메모리 오버플로와 누수 문제를 해결해야하는 상황이나 더 높은 동시성을 달성하는 데, 가비지 컬렉션이 방해되는 상황이 오면, 이 '자동화'된 기술을 적절히 모니터링하고 조율할 수 있어야 함.

## 3.2 대상이 죽었는가?
### 3.2.1 참조 카운팅 알고리즘
- 객체를 가리키는 참조 카운터를 추가하고, 참조갛는 곳이 늘어날 때마다 카운터 값을 1씩 증가
- 참조 하는 곳이 하나 사라질 때마다 카운터 값 1씩 감소
- 카운터 값이 0이 된 객체는 더이상 사용되지 않음
- 횟수를 세기 위한 추가 메모리 할당이 필요.
- 단점
	- 순환 참조 일 경우, 제대로 된 참조 카운팅을 알기가 쉽지 않음.
- JVM에서는 현재 쓰이고 있지 않다.
- 참조의 정의 (JDk 1.2부터)
	- 강한 참조: 프로그램 코드에서 참조를 할당할 때
		- 절대 회수 대상이 아님.
	- 부드러운 참조: 유용하지만 필수는 아닌 객체
		- OOM 나기 직전에 회수 대상 목록에 추가된다.
	- 약한참조: 연결 강도가 더 약함
		- 다음번 GC까지만 살아있다.
		- 메모리가 넉넉하더라도 약하게 참조된 객체는 모두 회수
		- `WeakReference` 클래스가 구현
	- 유령 참조: 
		- 객체 수명에 아무런 영향을 주지 않음.
		- 대상 객체가 회수될 때 알림을 받기 위한 용도,
		- `PhantomReference` 로 구현
	- 파이널 참조

### 3.2.2 도달 가능성 알고리즘
- 오늘날 주류 프로그래밍 언어들에 이용중.
- GC 루트,즉 시작노드부터 참조된 경로(참조 체인) 탐색.
	- GC 루트로부터 참조체인이 없다면, 객체는 더이상 사용할 수 없다.
- JVM에서 GC 루트로 이용할 수 있는 객체
	- 가상머신스택
	- 메서드영역에서 클래스가 정적 필드로 참조하는 객체
	- 메서드 영역에서 상수로 참조되는 객체
	- 네이티브 메서드 스택에서 JNI가 참조하는 객체
	- 자바 가상 머신 내부에서 ㅆ이는 참조
	- 동기화 락으로 잠겨있는 모든객체
	- 자바 가상머신 내부 상황을 반영하는 JMX에 등록된 콜백, 로직코드 캐시 등

### 3.2.4 살았나 죽었나? -> 유예 단계가 한 번 있음.
- 도달 불가능으로 판단한 객체는 두번의 marking 과정을 거쳐야 한다.

## 3.3 가비지 컬렉션 알고리즘
### 3.3.1 세대 단위 컬렉션 이론
- 가정
	- 약한 세대 가설: 대두사의 객체는 일찍 죽는다.
	- 강한 세대 가설: 가비지 컬렉션 과정에서 살아남은 횟수가 늘어날 수록 오래 살 가능성이 커진다.
	- 세대간 참조 가설: 세대간 참조의 개수는 같은 세대 안에서의 참조보다 훨씬 적다.
- 자바 힙을 여러 영역으로 나누어 가비지 컬렉터는 한 번에 하나 또는 몇 개 영역만 선택해 회수할 수 있는데 이를 기준으로 마이너GC, 메이저GC, 전체 GC 식으로 부르곤 한다.
- 자바 힙을 신세대, 구세대로 나눈다. 따로 GC한다.
- 구세대를 모두 스캔하는 것은 비용이 많이 듦으로
	- 신세대에 기억 집합이라는 전역 데이터 구조를 두고, 구세대를 작은 조걱 몇 개로 나누고 그 중 어느 조각에 세대간 참조가 있는지 기록해 관리한다.
	- 마이너 GC일때, 이 세대간 참조를 포함하는 작은 메모리 블록 안의 객체들만 GC루트에 추가하여 스캔.
	- 객체 사이에서 참조 관계 변화를 정확하게 관리해야한다.
- 용어 정리
	- 부분 GC: 자바 힙의 일부만 회수하는 가비지 컬렉션
		- 마이너 GC
		- 메이저 GC
		- 혼합 GC
	- 전체 GC

### 3.3.2 마크-스윕 알고리즘
- 알고리즘 작업을 표시와 쓸기라는 두 단계로 나눠 진행한다.
- 단점
	- 실행 효율이 일정하지 않다.
	- 메모리 파편화가 심하다.
### 3.3.3 마크-카피 알고리즘
- 마크-스윕 알고리즘의 단점인, 회수할 객체가 많아질 수록 효율이 떨어지는 문제를 해결하기 위해 제안됨.
- 가용 메모리를 똑같은 크기의 두 블록으로 나눠 한 번에 한 블록만 사용.
	- 대다수가 회수된다면 생존한 소수의 객체만 복사하면 된다.
- 단점
	- 가용 메모리를 절반으로 줄여 낭비가 제법 심하다.
- 오늘날 상용 자바 가상머신 대부분은 신세대에 이 알고리즘을 활용한다.
	- 신세대 객체중 98%가 첫 번째 가비지 컬렉션에서 살아남지 못한다. -> 신세대용 메모리 영역을 1:1로 나눌 필요가 없다. -> 아펠스타일 컬렉션
- 아펠스타일 컬렉션: 핫스팟 가상 머신의 신세대 컬렉터를 모두 신세대 메모리의 레이아웃을 이 전략에 부합되게 구성했다.
	1. 신세대를 하나의 큰 에덴 + 두개의 작은 생존자 공간으로 나눔.
	2. 메모리를 할당할 때는 생존자 공간 중 하나와 에덴만 사용.
	3. 가비지 컬렉션이 시작되면 에덴과 생존자 공간에서 살아남은 객체들을 나머지 생존자 공간으로 하나씩 복사
	4. 에덴과 이전 생존자 공간을 곧바로 비운다.
	- 기본적으로 에덴 8, 생존자 구역 1 * 2개 로 구성한다.
		- 만약 생존자구역이 부족하다면, 구세대를 활용해 메모리 할당을 보증한다.
### 3.3.4 마크-컴팩트 알고리즘
- 마크-카피 알고리즘의 경우, 객체 생존율이 높을 수록 효율이 나빠진다.
	- 구세대에는 적합하지 않다.
- 구세대 객체들의 생존 특성을 감안하여 고안됨.
	1. 표시 단계는 마크-스윕과 같다.
	2. 컴팩트 단계에서 생존한 모든 객체를 메모리 영역의 한쪽 끝으로 모은다.
	3. 나머지 공간을 한꺼번에 비운다.

- 도구들의 범주
	- 상용 인증 도구
	- 공식 지원 도구
	- 실험적 도구

	- 개발자는 JDK 도구 라이브러리의 인터페이스와 구현 코드를 이용하여 강력한 모니터링과 분석 기능을 애플리케이션에서 직접 제공할 수 있다.
### 4.2.1 jps: 가상 머신 프로세스 상태 도구
- `ps` 와 같은 역할
- 동작중인 가상머신 프로세스 목록을 조회
	- 가상 머신이 실행한 메인 클래스 이름
	- 로컬 가상머신 식별자 (=LVMID)
- 원격 가상머신의 프로세스 상태도 알아낼 수 있다.
### 4.2.2 jstat: 가상 머신 통계 정보 모니터링 도구
- JVM 통계적 모니터링 도구
- 가상 머신의 다양한 작동 상태 정보를 모니터링
	- 클래스 로딩
	- 메모리
	- GC
	- JIT 컴파일 등 런타임 데이터
- `interval`, `count`로 간격과 횟수를 설정하여 여러번 실행가능
- 가상 머신 상태 정보를 텍스트로만 보여주어, GUI보다는 직관적이지 않다.
	- 다만 콘솔 환경에서 사용하기 좋다.
### 4.2.3 jinfo: 자바 설정 정보 도구
- 가상 머신의 다양한 매개 변수를 실시간으로 확인하고 변경하는 도구
- 가상 머신 시작시 명시한 매개 변수 목록을 보고 싶을 때, `-flag` 사용
- 가상 머신 시작시 명시하지 않았지만, 적용된 값을 보고 싶다면 `-flag` 옵션 다음에 원하는 매개변수 이름을 넣어 사용.
- JDK6부터 실행중에 일부 매개 변수의 값을 수정할 수 있게되었다.
### 4.2.4 jmap: 자바 메모리 매핑 도구
- 힙 스냅숏을 파일로 덤프
- 다른 방식도 있다.
	- `-XX:+HeapDumpOnOutOfMemoryError` 매개변수 등
- jmap 으로 F-큐, 자바 힙과 메서드 영역 상세 정보도 알 수 있다.
### 4.2.5 jhat: 가상 머신 힙 덤프 스냅숏 분석 도구
- JDK 8까지 제공되던 JVM 힙 분석 도구. jmap 으로 덤프한 힙 스냅숏을 분석 가능하다.
- JDK 9 부터 jhsdb가 대신한다.
- 잘 사용하지 않는 이유
	- 힙 스냅숏 덤프를 애플리케이션이 배포된 서버에서 직접 분석하는 일은 드물다.
	- 다른 전문 분석 도구가 기능면에서 jhat 보다 뛰어나다.
### 4.2.6 jstack: 자바 스택 추적 도구
- 현재 가상 머신의 스레드 스냅숏을 생성
	- 현재 가상 머신에서 실행중인 각 스레드의 메서드 스택 집합
- 주로 스레드가 장시간 멈춰있을 때 원인을 찾기 위해 생성
	- 스레드 사이에 교착 상태 deadlock, 무한루프, 외부 자원 요청으로 인한 긴 기다림을 파악하고자함.
- 각 스레드의 우선순위, CPU 사용시간, 생성 후 경과 시간, 스레드 상태, 메서드 호출 스택 등 다양한 정보를 알 수 있다.
- JDK5 부터 `Tread.getAllStackTraces()`메서드를 이용하여 가상 머신이 실행하는 모든 스레드의 StackTraceElement 객체를 얻을 수 있다.

### 4.4.1 HSDIS
- 기술이 발전하면서 고성능 가상 머신의 실제 구현은 '자바 가상 머신 명세'의 정의와 조금씩 차이가 생기기 시작했다.
	- 자바 가상 머신 명세는 점차 개념모델이 되어갔다. 즉, 결과가 명세의 설명과 똑같다면 실행 방식은 달라도 된다.
- 프로그램 실행의 '의미'를 논할 때는 바이트코드 수준에서 분석하는 것이 합리적이다.
	- JIT 컴파일러가 엄청난 양의 실행 코드를 동적으로 생성해 코드 캐시에 추가하는데, 이 환경에서 쉽게 디버깅할 간단한 방법이 없다. 그래서 문제를 우회하는 방식 -> HSDIS
- 오라클이 권장하는 핫스팟 가상 머신 JIT 컴파일 코드용 디스어셈블리 플러그인
### 4.4.2 JITWatch
- 핫스팟 JIT 컴파일러용 로그 분석 및 시각화 도구. HSDIS와 주로 함께 쓰인다.

## 4.5 마치며
- '자바 가상 머신 명세'자체는 모니터링이나 문제 해결 방법을 정의하고 있지 않으므로 도구에 따라 지원하는 가상 머신과 기능 범위가 다를 수 있다.

# 6장. 클래스 파일 구조
## 6.1 들어가며
- 오랜 세월이 흘렀음에도 컴퓨터는 여전히 0과 1만 인식한다.
- 가상머신이 출현하면서, 프로그램을 네이티브 코드로 컴파일하지 않아도되는 길이 생겼다.
## 6.2 플랫폼 독립을 향한 초석
- 앞으로도 오랫동안 다양한 하드웨어 아키텍처와 운영체제가 공존할 것이며, '플랫폼 독립적' 이라는 이상은 결국 운영체제 위의 애플리케이션 계층에서 실핸될 것이다.
- 자바 가상 머신이 다양한 플랫폼을 지원하고, 모든 가상 머신이 동일한 프로그램 저장 방식 ByteCode 를 지원한다는 사실이 플랫폼 독립성의 핵심.
- 최근에는 자바 가상 머신이 제공하는 또 하나의 독립성인 '언어 독립성'이 개발자 사이에서 점점 더 중요하게 여겨지고 있다.
	- ==!! 언어 독립성이 최근 가장 중요하다는 것이 너무 공감됨.==
- 자바 기술은 초기 설계부터 가상 머신에서 다른 언어를 실행할 가능성을 염두에 두었다.
	- **자바 가상 머신은 자바를 포함하여 어떠한 프로그래밍 언어에도 종속되지 않는다.**
- 언어 독립성을 보장하는 핵심
	- 가상 머신과 바이트코드 저장 형식
	- 클래스파일이라는 특정한 바이너리 파일 형식에만 의존한다.
- 클래스파일
	- 자바 가상 머신 명령어 집합과 심벌 테이블 등을 포함한다.
	- 자바 가상 머신 명세에서 필수 구문과 특정 구조를 따르도록 클래스 파일을 제약한다.
- 바이트코드 형식은 튜링완전 Turing complete 하기 때문에 자바 가상 머신에서 어떠한 언어도 표현할 수 있도록 보장.
	- ==?: 튜링 완전이 무엇이지?==
```text
[Java]    [Kotlin]    [Scala]    [Groovy]
   |          |           |           |
   +----------+-----------+-----------+
                      |
               각 언어 컴파일러
                      |
               JVM 바이트코드(.class)
                      |
                     JVM
                      |
         +------------+-------------+
         |            |             |
   클래스 로딩     바이트코드 검증    실행 엔진(JIT/GC)
                      |
                 OS / CPU
```
- 자바 컴파일러를 사용하면 자바 코드를 바이트코드로 바궈 '클래스파일'로 저장할 수 있고, 코틀린 등 다른 언어의 컴파일러 역시 해당 언어의 소스코드를 '클래스 파일'로 컴파일 가능.
## 6.3 클래스 파일의 구조
- **자바 기술이 하위 호완성이 매우 좋은데, 그 이유는 '클래스 파일 구조가 안정적'이기 때문**
	- 클래스 파일의 기본 구조는 거의 변하지 않았다.
	- 기본적으로 원래 구조를 유지한 채 내용을 추가하거나 기능을 확장하는 형태.
- 모든 클래스 파일은 하나의 클래스 혹은 인터페이스를 정의한다.
- 동적으로 생성하여 클래스 로더에 직접 제공하거나, 파일로 저장해둘 수 있다.
- 클래스 파일에 데이터를 저장하는 데에, 의사구조 pseudo structure 를 이용한다.
	- 부호 없는 숫자 unsigned number: 데이터 타입 표현
	- 테이블: 계층적으로 구성된 복합 구조의 데이터를 설명.
- 클래스파일은 구분자가 없다. 단, 바이트 순서, 의미, 길이 가 모두 엄격하게 제한되며 변경할 수 없다.
### 6.3.1 매직 넘버와 클래스 파일 버전
- 매직넘버: 모든 클래스 파일의 처음 4byte
	- JVM이 허용하는 클래스 파일인지 여부를 빠르게 확인.
	- ==!!: 클래스파일은 0xCAFEBABE 가 매직넘버 ㅋㅋㅋ 귀엽다==
- 버전: 매직넘버 다음 4byte, 메이저버전. 마이너버전
	- 상위 버전 JDK는 하위 버전을 인식가능하지만, 하위버전은 상위 버전을 인식할 수 없다.
	- 자바 가상 머신 명세에서 금지해둠. -> 안정성을 위해서?!
### 6.3.2 상수풀
- 클래스 파일엥서 가장 먼저 등장하는 테이블 타입 데이터 항목
- 유형
	- 리터럴: 자바 언어 수준에서 이야기하는 상수와 비슷한 개념
	- 심벌참조: 컴파일과 관련된 개념
		- 모듈에서 참조하는 패키지
		- 클래스와 인터페이스의 완전한 이름
		- 필드 이름과 서술자
		- 메서드 이름과 서술자
		- 메서드 핸들과 메서드 타입
		- 동적으로 계산되는 호출 사이트와 동적으로 계산되는 상수
- 자바에서 링크는 가상 머신이 클래스파일을 로드할 때 동적으로 이루어진다.
	- 클래스파일을 로드할 때, 상수 풀에서 해당 심벌 참조들을 가져온다. 그 다음 클래스가 생성되거나 구동할 때 해석하여 실제 메모리 주소로 변환한다.
### 6.3.3 접근 플래그
- 상수 풀 다음 2byte는 현재 클래스의 접근 정보를 식별하는 접근 플래그.
	- public여부
	- final 여부
	- invokespecial 허용 여부
	- 인터페이스 여부
	- 추상클래스나 인터페이스인지 여부
	- 컴파일러가 자동생성했는지 여부
	- 애너테이션 여부
	- enum 여부
	- 모듈 여부
### 6.3.4 클래스 인덱스, 부모 클래스 인덱스, 인터페이스 인덱스
- 클래스 파일의 상속 관계를 규정.
- 클래스 인덱스, 부모 클래스 인덱스는 현재 클래스와 부모 클래스의 완전한 이름을 결정하는 데에 쓰인다.
- 인터페이스 인덱스 컬렉션은 현재 클래스가 구현한 인터페이스들을 기술.
### 6.3.5 필드 테이블
-

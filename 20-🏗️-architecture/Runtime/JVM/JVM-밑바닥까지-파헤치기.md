## 2.2 런타임 데이터 영역
- 자바 프로그램을 실행하는 동안 필요한 메모리를 몇 개의 데이터 영역으로 나눠 관리
- 각 영역들은 목적/생성시점/삭제시점이 있다.

| ----------- | ------------------------------------------------------------ | --------------------- | --------------------- | ---------------- |
| 프로그램 카운터    | 현재 실행 중인 JVM 바이트코드의 **명령어 주소 저장**. 스레드가 어디까지 실행했는지 추적        | 스레드 생성 시              | 스레드 종료 시              | **스레드 프라이빗 메모리** |

### 프로그램 카운터
- 현재 실행중인 스레드의 바이트코드 줄 번호 표시기
- 프로그램 제어 흐름, 분기, 순환, 점프 등을표현한다.
- JVM에서 멀티스레딩은 CPU 코어를 여러 스레드가 교대로 사용하는 방식으로 구현되어있다.
	- 특정 시각에 각 코어는 한 스레드의 명령어만 실행하게 된다.
	- 즉, 스레드 전환 후 이전에 실행하다 멈춘 지점을 정확하게 복원하려면 각 스레드는 고유한 프로그램 카운터가 필요하다.
- 스레드 프라이빗메모리
	- 스레드카운터는 서로 영향을 주지 않는 독립된 영역에 저장된다.
- 자바 메서드를 실행중일때는, 실행 중인 바이트코드 명령어의 주소가 프로그램 카운터에 기록됨.
- 네이티브메서드를 실행중일때는 프로그램카운터값은 `Undefined` 입니다.

### 자바 가상 머신 스택
- 스레드 프라이빗하다
- 연결된 스레드와 수명주기를 같이한다.
- 각 메서드가 호출될때마다 JVM은 스택 프레임을 만들어, 정보를 저장하고 스택 프레임을 가상머신 스택에 push 하고 메서드가 끝나면 pop 한다.
	- 지역변수테이블
	- 피연산자 스택
	- 동적 링크
	- 메서드 반환값 등의 정보
- C, C++ 프로그램의 메모리 구조보다 복잡하게 되어있다.
- 지역변수테이블
	- 기본 데이터 타입, 객체 참조, 반환 주소 타입을 저장한다
	- 이 데이터 타입들을 저장하는 공간을 지역변수슬롯 이라고 한다.
		- 이 슬롯은 기본 32bit. double 타입과같이 길이가 64bits 라면 슬롯을 2개 할당한다.
	- 컴파일 타임에 지역변수 테이블을 구성하는데 필요한 데이터 공간(즉, 슬롯 갯수)을  모두 할당하여 결정한다.
- 스택 메모리 영역에서는 두 가지 오류가 발생할 수 있다.
	- StackOverflowError: 스택 깊이가 가상 머신이 허용하는 깊이보다 클 경우
	- OutOfMemoryError: 스택 용량을 동적으로 확장할 수 있는 경우에, 스택을 확장하려는 시점에 여유 메모리가 충분하지 않을 경우.

### 메서드 영역
- 모든 스레드가 공유
- JVM이 읽어들인 타입 정보, 상수, 정적 변수, JIT 컴파일러가 컴파일한 코드 캐시 등을 저장하는 데 이용
- 이전에는 영구세대 영역에 메서드 영역을 저장했지만, 최근들어 점진적으로 네이티브 메모리에 구현
	- 메서드 영역을 영구 세대에 구현하면 자바 애플리케이션들이 메모리 오버플로를 겪을 가능성이 커졌다.
- JVM spec 에서는 메서드 영역에 제약을 거의 두지 않았다.
	- 연속될 필요가 없고, 크기를 고정할 수 있고-확장할 수도 있음.
	- 가비지 컬렉션을 할필요가 별로 없음 (회수 효과가 매우 작기 때문)
- 메모리를 더이상 할당할 수 없다면 OOM

### 런타임 상수풀
- 동적이다.
- 메서드 영역의 일부
- 클래스 버전, 필드, 메서드, 인터페이스 등 클래스 파일에 포함된 설명 정보에 더해 컴파일타임에 생성된 다양한 리터럴과 심벌 참조가 저장된다.
	- 클래스를 로드할 때 생성된 다양한 리터럴과 심벌 참조가 저장된다.

### 다이렉트 메모리
- 가상 머신 런타임에 속하지 않는다.
- 하지만 JDK 1.4에서 NIO가 조입되면서 채널과 버퍼기반 I/O 메서드가 소개됨
	- 힙이 아닌 메모리를 직접 할당할 수 있는 '네이티브 함수 라이브러리'를 이용하며 이 메모리에 저장되어있는 DirectByteBuffer 객체를 통해 작업을 수행
		- 자바힙과 네이티브 힙 사이에 데이터를 복사해 주고받지 않아도 되어서 일부 시나리오에서 성능을 크게 개선함.
- 물리 메모리를 직접 할당하기 때문에 자바 힙 크기의 제약과 무관하지만, 하부 기기의 총 메모리 용량과 프로세서가 다룰 수 있는 주소 공간을 넘어설 수 없으므로 주의해야한다.
- 사용되는 모든 메모리영역의 합이 물리메로리 한계를 넘어서면 동적 확장을 시도할때 OOM이 발생한다.

## 핫스팟 가상머신에서의 객체 들여다보기
### 객체 생성
1. JVM이 new 명령에 해당하는 바이트 코드 만남
2. 이 명령의 매개변슈가 상수 풀안의 클래스를 가리키는 심벌참조인지 확인
3. 클래스를 가리키고 있다면, 클래스가 로딩-resolve-초기화 되었는지 확인
	1. 만약 안되어있다면 로딩부터 한다
4. 새 객체를 담을 메모리를 자바 힙에 할당
	1. 클래스를 로딩하고나면 어떤 크기가 필요한지 완벽하게 알 수 있다
	2. 사용중인 메모리는 한쪽에, 여유 메모리는 반대편에. 경계에는 포인터가 있다.
	3. 할당하면, 포인터를 객체 크기만큼 이동시킨다 = 포인터 밀치기
	4. 자바 힙이 규칙적이지 않으므로, JVM은 가용 메모리 블록들을 목록으로 관리하여, 객체를 저장할 수 있는 충분한 공간을 찾아 할당 후, 목록 갱신 (이 목록을 free list 라고함.)
	5. 자바 힙이 규칙적인지의 여뷰, GC가 컴팩트를 할 수 있느냐에 따라 다른 방식을 취하기도 함.
5. 할당이 종료되면, JVM은 할당 받은 공간을 데이터 0으로 초기화.
6. JVM은 각 객체에 필요한 설정을 각 객체의 객체 헤더에 저장
	1. 어느 클래스의 인스턴스인지?
	2. 메타 정보는 어떻게 찾는지
	3. 해시코드
	4. GC age
7. 이제 자바 프로그램 관점에서, 생성자 `<init>()` 를 실행하여 개발자 의도대로 초기화

```text
<자바코드>     <바이트코드>
new ---------> new // JVM 관점에서 객체셍성
    ---------> invokespecial // 자바 프로그램 관점에서 객체 생성
```

- 멀티스레딩 환경에서 동시성 문제 해결방식
	- 방법1. 메모리 할당을 동기화하거나
	- 방법2. 스레드마다 다른 메모리 공간을 할당함. 이를 스레드로컬할당버퍼(TLAB) 라고한다.

### 객체에 접근하기
- 자바 프로그램은 스택에 있는 참조 데이터를 통해 힙에 들어 있는 객체들에 접근해 이를 조작
- JVM Spec 은 참조 타입을 단지 '객체를 가리키는 참조' 라고만 정했을 뿐, 구체적인 방식 규정은 없다.
- 주로 핸들 방식, 다이렉트 포인터를 사용해 구현
- 핸들
	- 참조에 '안정적인' 핸들의 주소가 저장됨.
	- 객체의 위치가 바뀌는 상황에서도 참조 자체는 손댈 필요가 없음.
	- 핸들 내의 인스턴트 데이터 포인터만 변경
- 다이렉트 포인터
	- 자바 힙에 위치한 객체에서 인스턴스 데이터 뿐 아니라 타입 데이터에 접근하는 길도 제공
	- 스택에 참조에는 객체 주소가 그대로 저장됨
	- 속도가 빠름. -> 실행 시간에 유리

## 실전: OutOfMemoryError 예외
### 자바 힙 오버플로
- 원인: 객체를 계속 생성하고 그 객체들에 접근할 경로가 살아있는 경우
- ==!: p.78 힙덤프 사례를 볼 수 있음.==
- 대응 방법
	- 메모리 누수
		- 누수된 객체로부터 GC 루트까지의 참조 사슬을 살펴본다.
		- 왜 가비지 컬렉터가 회수 못했는지 확인
	- 만약 메모리 누수가 아닐 경우
		- 더 많은 메모리를 할당할 수 있는지 확인
		- 햅 매개변수설정 `-Xmx`, `-Xms`
		- 코드에서 수명 주기가 너무 길거나 상태를 너무 오래 유지하는 객체 확인
		- 공간 낭비가 심한 데이터 구조를 쓰고 있는지 확인

### 가상머신 스택과 네이티브 메서드 스택 오버플로
- 예외 발생 원인
	- 스레드가 요구하는 스택 깊이가 가상 머신이 허용하는 최대 깊이보다 크면 stackoverflowError
	- 가상 머신이 스택 메모리를 동적으로 확장하는 기능을지원하나, 가용 메모리가 부족해 스택확장이 불가능할경우 OOM
	- 핫스팟에서도 스레드를 계속 만들면 메모리 오버플로를 일으킬 수 있다.
- 보통 운영체제가 각 프로세스에 할당하는 메모리크기가 제한적임.

### 메서드 영역과 런타임 상수 풀 오버플로
- JDK7 부터 상수풀을 자바힙으로 옮김

## 3.2 대상이 죽었는가?
### 3.2.1 참조 카운팅 알고리즘
- 객체를 가리키는 참조 카운터를 추가하고, 참조갛는 곳이 늘어날 때마다 카운터 값을 1씩 증가
- 참조 하는 곳이 하나 사라질 때마다 카운터 값 1씩 감소
- 카운터 값이 0이 된 객체는 더이상 사용되지 않음
- 횟수를 세기 위한 추가 메모리 할당이 필요.
- 단점
	- 순환 참조 일 경우, 제대로 된 참조 카운팅을 알기가 쉽지 않음.
- JVM에서는 현재 쓰이고 있지 않다.
- 참조의 정의 (JDk 1.2부터)
	- 강한 참조: 프로그램 코드에서 참조를 할당할 때
		- 절대 회수 대상이 아님.
	- 부드러운 참조: 유용하지만 필수는 아닌 객체
		- OOM 나기 직전에 회수 대상 목록에 추가된다.
	- 약한참조: 연결 강도가 더 약함
		- 다음번 GC까지만 살아있다.
		- 메모리가 넉넉하더라도 약하게 참조된 객체는 모두 회수
		- `WeakReference` 클래스가 구현
	- 유령 참조: 
		- 객체 수명에 아무런 영향을 주지 않음.
		- 대상 객체가 회수될 때 알림을 받기 위한 용도,
		- `PhantomReference` 로 구현
	- 파이널 참조

### 3.2.2 도달 가능성 알고리즘
- 오늘날 주류 프로그래밍 언어들에 이용중.
- GC 루트,즉 시작노드부터 참조된 경로(참조 체인) 탐색.
	- GC 루트로부터 참조체인이 없다면, 객체는 더이상 사용할 수 없다.
- JVM에서 GC 루트로 이용할 수 있는 객체
	- 가상머신스택
	- 메서드영역에서 클래스가 정적 필드로 참조하는 객체
	- 메서드 영역에서 상수로 참조되는 객체
	- 네이티브 메서드 스택에서 JNI가 참조하는 객체
	- 자바 가상 머신 내부에서 ㅆ이는 참조
	- 동기화 락으로 잠겨있는 모든객체
	- 자바 가상머신 내부 상황을 반영하는 JMX에 등록된 콜백, 로직코드 캐시 등

### 3.2.5 메서드 영역 회수하기
